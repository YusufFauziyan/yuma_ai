# Yuma AI — Backend

Express.js API server for Yuma AI chatbot with Google Gemini integration, MySQL persistence, and user authentication.

## Tech Stack

- **Runtime**: Node.js
- **Framework**: Express.js
- **AI**: Google Gemini API (`@google/generative-ai`)
- **Database**: MySQL
- **ORM**: Prisma v7 (`@prisma/client`, `@prisma/adapter-mariadb`)
- **Auth**: Token-based middleware (compatible with Auth.js JWT)
- **Streaming**: Server-Sent Events (SSE)

## Getting Started

### 1. Install Dependencies

```bash
npm install
```

### 2. Configure Environment

Copy the example and fill in your values:

```bash
cp .env.example .env
```

```env
DATABASE_URL="mysql://root:password@localhost:3306/yuma_ai"
GEMINI_API_KEY=your_gemini_api_key_here
PORT=5001
```

Get your Gemini API key from [Google AI Studio](https://aistudio.google.com/apikey).

### 3. Setup Database (MySQL & Prisma)

Ensure you have MySQL running and create the database:
```sql
CREATE DATABASE yuma_ai;
```

Run Prisma migrations to initialize the schema:
```bash
npx prisma migrate dev --name init
```

Generate the Prisma Client:
```bash
npx prisma generate
```

### 4. Run Development Server

```bash
npm run dev
```

The server starts at `http://localhost:5001`.

## Database Management (Prisma CLI)

We use Prisma v7 with Driver Adapters for database operations. You can run these commands via `npx` or using the NPM scripts defined in `package.json`.

| Script | Command | Description |
|--------|---------|-------------|
| `npm run prisma:migrate` | `npx prisma migrate dev` | Apply migrations to dev database |
| `npm run prisma:generate` | `npx prisma generate` | Regenerate the Prisma Client |
| `npm run prisma:studio` | `npx prisma studio` | Open the visual database editor |
| `npm run prisma:reset` | `npx prisma migrate reset` | Drop database, recreate, and run migrations |

## API Endpoints

All endpoints (except `/health`) require an `Authorization: Bearer <token>` header.

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/health` | Health check (no auth) |
| `POST` | `/api/chat` | Stream AI response via SSE |
| `GET` | `/api/conversations` | List user's conversations |
| `POST` | `/api/conversations` | Create a new conversation |
| `GET` | `/api/conversations/:id` | Get conversation with messages |
| `DELETE` | `/api/conversations/:id` | Delete a conversation |

### Example: Health Check

```bash
curl http://localhost:5001/health
# {"status":"ok","timestamp":"2026-04-29T05:00:00.000Z"}
```

### Example: Create Conversation (Authenticated)

```bash
curl -X POST http://localhost:5001/api/conversations \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{"title": "My Chat"}'
```

### Example: Stream Chat Response

```bash
curl -X POST http://localhost:5001/api/chat \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{"conversationId": "<id>", "message": "Hello!"}'
```

Response is an SSE stream:

```
data: {"text":"Hello","conversationId":"abc-123"}
data: {"text":"! How can","conversationId":"abc-123"}
data: {"text":" I help you?","conversationId":"abc-123"}
data: {"done":true,"conversationId":"abc-123"}
```

## Database Schema

```prisma
model User {
  id, name, email (unique), image?, createdAt
  → has many Conversations
}

model Conversation {
  id, userId (FK→User), title, createdAt, updatedAt
  → has many Messages (cascade delete)
}

model Message {
  id, conversationId (FK→Conversation), role, content (TEXT), createdAt
}
```

- Data is isolated by `userId` in all operations.
- Messages cascade delete when a conversation is deleted.

## Authentication

The backend expects a Base64-encoded JSON token in the `Authorization` header. This token is generated by the frontend from the Auth.js session:

```json
{
  "email": "user@example.com",
  "name": "User Name",
  "picture": "https://...",
  "sub": "user-id",
  "iat": 1700000000
}
```

The middleware decodes this, finds or creates the user via Prisma (`findOrCreateUser`), and attaches `req.user` to every request. All conversation operations are scoped by `req.user.id`.
